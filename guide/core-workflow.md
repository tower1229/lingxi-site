# 核心工作流

灵犀提供一套**可伸缩的开发工作流**，覆盖从需求到交付的完整生命周期。每个环节都是可选的，你可以根据任务复杂度灵活组合。

## 工作流总览

```
/task  →  /vet  →  /plan  →  /build  →  /review
 必选       可选          可选       可选        推荐
```

::: tip 设计哲学
灵犀是**工具包**，不是流水线。除了 `/task` 作为需求起点外，其余环节均可跳过。选型责任在你。
:::

## /task — 创建任务

一切从需求开始。`/task` 是灵犀工作流的起点，也是唯一的必选步骤。

```
/task <需求描述>
```

**示例：**
```
/task 添加用户登录功能，支持邮箱和手机号登录
/task 优化首页加载性能，目标首屏时间 < 1s
```

**灵犀会做什么：**
- 自动生成任务编号（001, 002...）和标题
- 创建结构化的任务文档：`.cursor/.lingxi/tasks/001.task.<标题>.md`
- 引导你完成**需求提纯**：分析、放大、确认

任务文档是整个工作流的核心载体，后续所有命令都围绕它展开。

## /vet — 审查 task 文档（可选）

对任务文档进行多维度审查，提升需求质量。

```
/vet [taskId]
```

- 可以多次执行，迭代优化
- 不产出文件，仅在对话中给出审查结果和建议
- 省略 `taskId` 时自动使用最新任务

**审查维度：** 完整性、一致性、可行性、边界条件等。

## /plan — 任务规划（可选）

基于任务文档生成详细的执行计划和测试用例。

```
/plan [taskId]
```

**产出：**
- 任务规划文档：`.cursor/.lingxi/tasks/001.plan.<标题>.md`
- 测试用例文档

**适用场景：** 复杂任务、涉及多模块改动、需要明确执行步骤时使用。简单任务可跳过。

::: tip
`/plan` 可以配合 Cursor 的 Plan 模式使用，两者互补。
:::

## /build — 执行构建（可选）

根据任务文档和（可选的）规划文档，执行代码实现。

```
/build [taskId]
```

**两种执行模式：**

| 模式 | 条件 | 行为 |
|------|------|------|
| Plan-driven | 存在 plan 文档 | 按计划结构化执行（推荐） |
| Task-driven | 无 plan 文档 | Agent 基于 task 文档自行决策 |

::: tip
当使用 Cursor 的 Plan 模式时，也可以直接使用其内置的 build 功能，跳过灵犀的 `/build` 命令。
:::

## /review — 审查交付（推荐）

对完成的代码进行全方位审查，生成审查报告。

```
/review [taskId]
```

**核心审查维度：**
- 功能完整性
- 测试覆盖率
- 架构合理性
- 可维护性
- 回归风险

**按需审查维度：**
- 文档一致性
- 安全性
- 性能
- 端到端测试

## 流程示例

### 简单任务（快速通道）

```
/task 修复登录页面的表单验证 bug
/build
/review
```

### 复杂任务（完整流程）

```
/task 重构用户权限系统，支持 RBAC 模型
/vet 001
/plan 001
/build 001
/review 001
```

### 最小流程

```
/task 更新 README 中的安装说明
/build
```
